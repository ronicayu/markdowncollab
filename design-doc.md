## Problem Statement

In the agentic AI era, enterprise teams need collaboration tools where both humans and AI agents are first-class participants. Current enterprise tools (Confluence, Microsoft 365) store data in proprietary formats that AI agents can't natively read or write. Teams build fragile custom pipelines to sync between markdown (which both humans and LLMs are fluent in) and these enterprise tools. The round-trip — write in markdown, sync to Confluence, convert back to markdown for AI, push changes back — is lossy, doesn't support real-time collaboration, and breaks constantly.

The solution is not better sync. It's building collaboration tools that are markdown-native from the ground up, so AI agents can participate as peers without translation layers.

## Demand Evidence

- Teams observed building custom sync pipelines: markdown managed in git, synced to Confluence for team visibility, then converted back to markdown locally to invoke AI tools. Neither direction supports real-time collaboration.
- Open source tools (mark, confluence-markdown-sync, markdown-to-confluence) exist to address the sync pain but are one-directional utilities, not collaboration tools. The existence of 5+ sync tools confirms the pattern is widespread.
- Enterprise teams are locked into Confluence/Microsoft with "no access to other tools" — the pain is institutional, not individual. Engineering leads watch productivity drain with no ability to fix it.
- Stage: pre-product. No paying customers yet. Evidence is observational, not transactional.

## Status Quo

The current workaround in enterprise engineering teams:

1. Engineers write documentation in markdown, managed in git repositories
2. A sync pipeline (custom scripts, GitHub Actions, or tools like `mark`) pushes markdown to Confluence pages for team-wide visibility
3. PMs, product owners, and non-technical stakeholders read and comment in Confluence
4. When anyone needs AI help (drafting, editing, summarizing, reviewing), they convert Confluence content back to markdown locally
5. AI tools operate on the local markdown copy
6. Changes get pushed back through the pipeline

**What breaks:** Format translation is lossy (Confluence's storage format != markdown). Sync scripts are fragile and require maintenance. Real-time collaboration is impossible across the format boundary. Every AI interaction requires a manual round-trip. Comments and inline feedback in Confluence don't flow back to the markdown source.

**What it costs:** Engineering leads estimate significant productivity loss as teams context-switch between formats, maintain sync pipelines, and manually shuttle content between tools.

## Target User & Narrowest Wedge

**Target user:** Engineering lead at a mid-to-large enterprise company (100+ engineers). Locked into Confluence/Microsoft by IT policy. Watches team productivity drain because modern AI-native workflows don't work inside the approved tool stack. Gets promoted by improving team velocity and output quality. Gets in trouble when documentation falls behind or cross-team collaboration breaks down.

**Narrowest wedge:** A web-based real-time collaborative markdown editor where AI agents join as named participants with their own cursors. Humans and agents co-edit the same document simultaneously. No Confluence integration in v1 — just prove that real-time human+AI collaboration on markdown feels meaningfully different from copy-pasting ChatGPT output.

## Constraints

- Enterprise environments restrict tool adoption — IT approval, security review, procurement cycles
- Must work alongside (not immediately replace) existing Confluence/Microsoft stack
- Markdown must be the source of truth — no proprietary format
- AI agents must be first-class participants, not bolted-on features
- Human experience comes first in UX, but the collaboration protocol must be AI-native at the architecture level
- Bottom-up adoption strategy (engineering teams adopt first, then pull the org in — the Slack playbook)

## Premises

1. **Markdown is the right format for human+AI collaboration in enterprise.** Both humans and LLMs are natively fluent in markdown. Proprietary formats require translation layers that degrade quality.

2. **Enterprise teams will adopt a new tool rather than wait for Confluence/Microsoft to ship better AI.** Three reinforcing reasons: (a) incumbents move too slowly for transformative change, (b) proprietary formats are structurally wrong for AI — no amount of bolted-on AI features fixes that, (c) bottom-up adoption by engineering teams will force IT approval, as Slack did.

3. **The engineering lead is the right buyer.** They feel the pain most acutely and have influence over team tooling decisions, even in locked-down enterprises.

4. **Real-time collaborative markdown editing is the core product** — not a sync/bridge layer alongside existing tools.

## Approaches Considered

### Approach A: MVP Editor (CHOSEN)
Real-time collaborative markdown editor. Tiptap (ProseMirror-based) for the editor, Yjs CRDT for real-time sync via y-websocket. AI agents join as named participants with their own cursors. Humans and agents co-edit as peers. Inline suggestions rendered as diff blocks with accept/reject. Export to clean markdown.

- **Effort:** M (2-4 weeks human / 2-3 days CC+gstack)
- **Risk:** Medium — proves the core concept but doesn't address enterprise adoption yet
- **Pros:** Fastest validation. Answers the existential question. Clean architecture to build on.
- **Cons:** No enterprise story (SSO, permissions, Confluence integration). Cold-start adoption.

### Approach B: Confluence-Bridge Editor
Same editor + bidirectional Confluence sync from day one. Positioned as "the AI-native editing layer for your existing Confluence knowledge base."

- **Effort:** L (4-8 weeks human / 1-2 weeks CC+gstack)
- **Risk:** High — Confluence API complexity could eat the roadmap
- **Pros:** Removes switching cost argument. Engineering leads can adopt without IT approval battle.
- **Cons:** Plugin prison. Dependent on Atlassian's API. Sync bugs dominate year one.

### Approach C: Full Workspace
Complete Notion replacement — docs, specs, knowledge base, all markdown-native, all AI-agent accessible. Enterprise SSO, permissions, audit trail from day one.

- **Effort:** XL (3-6 months human / 3-4 weeks CC+gstack)
- **Risk:** High — scope creep, competing on feature surface area with well-funded incumbents
- **Pros:** Maximum differentiation. Full replacement story. Massive company if it works.
- **Cons:** Too long to validate. Building features before confirming demand.

## Recommended Approach

**Approach A: MVP Editor.** Answer the only question that matters first — does real-time AI collaboration feel meaningfully different from copy-pasting ChatGPT output? If yes, layer on Confluence sync (Approach B) as the enterprise wedge. If no, you saved months.

**Tech stack:**
- **Frontend:** Next.js + Tiptap (ProseMirror-based, markdown serialization built in)
- **Real-time sync:** Yjs CRDT with y-websocket server. Note: the basic server is ~50 lines, but production readiness requires room management, document persistence callbacks (Yjs state to SQLite on room idle), inactive room cleanup, and WebSocket reconnection handling. Budget for these in week 2.
- **AI agent integration:** REST endpoint that connects to the Yjs room as a named participant ("Agent: Claude"), reads the document, and applies edits via Yjs document API
- **AI backend:** Anthropic API (Claude) for initial agent capability — review and suggest edits
- **Persistence:** SQLite via Prisma, storing Yjs document state
- **Auth:** None in v1. Hardcode users and one agent. Note: the auth-less prototype is for internal concept validation only. Auth (at minimum magic link or OAuth) becomes a gate before any external user testing in week 3.

**Minimal data model:**
- `documents` — id, title, yjs_state (binary), created_at, updated_at
- `suggestions` — id, document_id, agent_name, original_text, suggested_text, status (pending/accepted/rejected/stale), created_at, resolved_at
- `sessions` — id, document_id, participant_name, participant_type (human/agent), connected_at, disconnected_at

**Core features (priority order):**
1. Two humans can open the same markdown doc and see each other's cursors and edits in real time
2. Button invites an AI agent into the doc; agent appears as a named cursor, reads the doc, inserts inline suggestions
3. Humans can accept/reject each agent suggestion with one click
4. Export to clean markdown file — exports the current document state with only accepted suggestions applied; pending suggestions are stripped, rejected suggestions are excluded

**AI suggestion model (core technical decision):**

Suggestions are implemented as custom ProseMirror node types, not raw CRDT operations. This keeps suggestions reversible without fighting Yjs's append-only semantics.

- A `suggestion` is a block-level ProseMirror node (not inline). It wraps one or more block nodes (paragraphs, headings, list items) with metadata: suggestion_id, agent_name, original_text, suggested_text, status (pending/accepted/rejected/stale). It renders as a visually distinct card containing a side-by-side or inline diff view (red strikethrough for removed, green highlight for added) with accept/reject buttons.
- When an agent proposes an edit, it inserts a `suggestion` node at the target location containing both the original and proposed content blocks
- Accept: replaces the `suggestion` node with the suggested_text content as normal document nodes
- Reject: replaces the `suggestion` node with the original_text content as normal document nodes
- Concurrent human edits to text inside a pending suggestion invalidate the suggestion (status -> stale). Detection: store a content hash of the original text when the suggestion is created. A Yjs observe callback on the suggestion node's content range fires on any remote update; if the content hash no longer matches, mark the suggestion as stale. Stale suggestions show a "Content changed — re-run?" prompt instead of accept/reject. The agent can be re-invited to re-evaluate.

**Agent lifecycle:**

1. User clicks "Invite Agent" button
2. Frontend sends POST to `/api/agent/invite` with document_id and prompt (e.g., "review this section")
3. Backend connects to the Yjs room as a named awareness participant ("Agent: Claude") — agent cursor appears in the editor
4. Backend reads the current Yjs document state, serializes to markdown
5. Backend calls Anthropic API with the markdown and the user's prompt
6. Backend parses the API response into suggestion operations, inserts `suggestion` nodes into the Yjs document
7. Backend disconnects from the Yjs room. Agent cursor disappears.
8. Agent operates in request-response mode only (no persistent monitoring in v1)

**Error handling:** If the Anthropic API is unavailable, rate-limited, or returns an error: the agent cursor shows a "failed" state (red indicator), a toast notification tells the user "Agent couldn't complete the request — try again," and the agent disconnects. No partial suggestions are inserted. Retry is manual (user clicks "Invite Agent" again). Timeout: 30 seconds.

**Explicitly skipped in v1:** Permissions, Confluence sync, file management, formatting toolbar, deployment polish, mobile, comprehensive tests.

**Effort timeline reconciliation:** The "48-hour prototype" from the cross-model perspective covers features 1-2 only (basic collab + agent inserts raw edits, no structured suggestion model). The full MVP (features 1-4 with the suggestion model, persistence, and error handling) is 2-3 days with CC+gstack / 2-4 weeks human. The 3-week success criteria timeline includes the MVP build (weeks 1-2) plus user validation (week 3).

## Blocking Decision

**Document model: AI-native protocol first, human UX on top.** The subagent's recommendation is correct. The Yjs CRDT document with markdown serialization IS the source of truth. The ProseMirror/Tiptap editor is a human-facing view layer. AI agents interact with the same Yjs document via the same CRDT protocol, not through a separate API. This means: the collaboration protocol is the product, the editor UI is one client among many. Future agent SDKs, CLI tools, and integrations all connect to the same Yjs room. This decision is made. Implement accordingly.

## Open Questions

1. **Agent interaction model:** Should AI agents only respond to explicit invitations ("Invite Agent" button), or should they be able to monitor and participate proactively? What are the UX implications of an agent that edits without being asked? (v1: invitation-only. Revisit after user testing.)

2. **Enterprise go-to-market timing:** When does Confluence integration become necessary? After 10 users? 100? What's the trigger to move from Approach A to Approach B?

3. **Competitive moat:** ColaMD, Nimbalyst, Ally, and mist are all in adjacent spaces. What specific feature or experience creates defensibility beyond "we're markdown-native"?

4. **Pricing model:** Per-seat? Per-agent? Usage-based on AI interactions? Enterprise contract?

## Success Criteria

- **Week 1:** Working prototype — two humans can co-edit a markdown document in real time with visible cursors
- **Week 2:** AI agent can join a document session, read content, and insert inline suggestions that humans can accept/reject
- **Week 3:** Demo to 5 engineering leads at enterprise companies. Measure: does at least 3/5 say "I want this for my team"?
- **Validation gate:** Before building Confluence sync (Approach B), get 3 LOIs or equivalent strong commitment signals from enterprise engineering leads

## Distribution Plan

- **v1 (MVP):** Deploy as a web app (Vercel or similar). Share URL directly with test users. No package manager, no binary distribution needed.
- **Future:** Self-hosted option for enterprise (Docker container). Potentially Confluence marketplace app if/when Approach B is pursued.
- **CI/CD:** GitHub Actions for build and deploy to hosting platform on merge to main.

## Dependencies

- Tiptap/ProseMirror ecosystem stability (well-maintained, low risk)
- Yjs CRDT library (battle-tested, used by many collaborative editors)
- Anthropic API availability and pricing for agent integration
- Access to 5+ enterprise engineering leads for validation interviews
