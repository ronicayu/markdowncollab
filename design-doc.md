## Problem Statement

In the agentic AI era, enterprise teams need collaboration tools where both humans and AI agents are first-class participants. Current enterprise tools (Confluence, Microsoft 365) store data in proprietary formats that AI agents can't natively read or write. Teams build fragile custom pipelines to sync between markdown (the natural lingua franca for both humans and LLMs) and these enterprise tools. The round-trip is lossy, doesn't support real-time collaboration, and breaks constantly.

The specific pain loop: write markdown in git, sync to Confluence for team visibility, convert Confluence back to markdown when AI help is needed, invoke AI, push changes back. Neither direction supports real-time collaboration.

## Demand Evidence

- Teams building custom sync pipelines between markdown-in-git and Confluence (observed in the wild)
- Multiple open source tools exist to solve parts of this (mark, confluence-markdown-sync, markdown-to-confluence) confirming widespread pain
- Engineering leads at enterprise companies report watching team productivity drain because AI-native workflows are impossible within their approved tool stack (Confluence/Microsoft only)
- The existence of ColaMD, Nimbalyst, Ally, and mist in 2026 confirms market pull toward AI-native markdown editing

## Status Quo

Teams duct-tape together markdown-in-git to Confluence sync pipelines. Open source tools like `mark` and `confluence-markdown-sync` provide one-directional sync but no real-time collaboration. When AI help is needed, someone converts Confluence pages back to local markdown, runs the content through ChatGPT or similar, then manually updates both sources. This process is:
- Lossy (formatting and metadata don't round-trip cleanly)
- Manual (someone maintains the sync scripts)
- Fragile (breaks when Confluence API changes or page structures diverge)
- Slow (no real-time collaboration in either direction)

## Target User & Narrowest Wedge

**Target user:** Engineering lead at a mid-to-large enterprise company. Locked into Confluence/Microsoft as approved collaboration tools. Watches their team lose 5-10 hours per week per engineer on format translation, copy-paste workflows, and maintaining sync pipelines. Gets promoted for shipping faster. Gets blamed when the team's velocity drops.

**Narrowest wedge:** A real-time collaborative markdown editor where AI agents join as named participants with their own cursors, reading and suggesting edits just like a human teammate. No Confluence integration yet. Just the core experience: does real-time AI collaboration feel meaningfully different from copy-pasting ChatGPT output?

## Constraints

- Must be web-based (enterprise teams need browser access, no desktop installs)
- Must support real-time collaboration (multiple humans + AI agents editing simultaneously)
- Markdown-native document format (not proprietary, not HTML-first)
- AI agents must appear as identifiable, named participants (not anonymous background processes)
- Agent suggestions must be reviewable and accept/reject-able (humans stay in control)

## Premises

1. **Markdown is the right format for human+AI collaboration in enterprise.** Agreed. Both humans and LLMs are fluent in it. Proprietary formats require translation layers that add friction and lose fidelity.

2. **Enterprise teams will adopt a new tool rather than wait for Confluence/Microsoft to ship better AI.** Defended with three reasons: (a) incumbents move too slow with incremental features, (b) proprietary formats are fundamentally wrong for AI and no amount of features fixes a structural problem, (c) bottom-up adoption by engineering teams will force IT approval, like Slack did.

3. **The engineering lead is the right buyer.** They feel the pain most acutely and can champion tool adoption within their org.

4. **Real-time collaborative markdown editing is the core product.** Not a sync layer, not a plugin. A new surface where humans and AI co-author as peers.

## Approaches Considered

### Approach A: Real-Time Collaborative Markdown Editor (MVP)
Web-based markdown editor with Yjs CRDT for real-time collab. AI agents join as named participants with cursors. Humans and agents co-edit as peers. Export to clean markdown. No Confluence integration yet.
- **Effort:** M (human: ~2-4 weeks / CC: ~2-3 days)
- **Risk:** Medium. Proves the core concept but doesn't address enterprise adoption yet.
- **Pros:** Fastest path to validation. Answers the existential question first. Clean architecture.
- **Cons:** No enterprise story yet. No integration with existing tools. Cold-start adoption challenge.
- **Reuses:** Tiptap + Yjs ecosystem (battle-tested), Anthropic API

### Approach B: Confluence-Bridge Editor (Enterprise Wedge)
Same editor with bidirectional Confluence sync from day one. Positioned as "the AI-native editing layer for your existing Confluence knowledge base."
- **Effort:** L (human: ~4-8 weeks / CC: ~1-2 weeks)
- **Risk:** High. Confluence sync complexity could consume the roadmap. API dependency risk.
- **Pros:** Removes switching cost argument. Easier enterprise adoption path.
- **Cons:** Plugin prison risk. Sync bugs dominate year one. Building ON Confluence, not replacing it.

### Approach C: Markdown-Native Workspace (Full Vision)
Full workspace: docs, specs, knowledge base. All markdown-native, all AI-agent accessible. Enterprise SSO, permissions, audit trail.
- **Effort:** XL (human: ~3-6 months / CC: ~3-4 weeks)
- **Risk:** High. Too long to validate. Capital intensive.
- **Pros:** Maximum differentiation. Full replacement story.
- **Cons:** Builds features before confirming demand.

## Recommended Approach

**Approach A: Real-Time Collaborative Markdown Editor (MVP)**

Rationale: Answer the only question that matters first. "Does real-time AI collaboration feel meaningfully different from copy-pasting ChatGPT output?" If yes, layer on Confluence sync (Approach B) as the enterprise wedge. If no, you saved months.

**Prototype positioning for enterprise users:** The MVP lacks auth, permissions, and Confluence integration. Position it as a hands-on sandbox: share a link with engineering leads, let them co-edit a doc alongside an AI agent for 15 minutes. The goal is to provoke the reaction, not to pass an IT security review. Enterprise features come after the core experience is validated.

**Tech stack:** Next.js + Tiptap (ProseMirror-based, markdown serialization built in) + Yjs CRDT with y-websocket server. A REST endpoint triggers a server-side Node.js process that maintains a Yjs WebSocket connection on behalf of the AI agent, reading document state and applying edits through the CRDT. Anthropic API for the agent brain.

**MVP features (priority order):**
1. Two humans can open the same markdown doc and see each other's cursors and edits in real time
2. Button that invites an AI agent who appears as a named cursor, reads the doc, and inserts inline suggestions (rendered as inline insertions with green highlight and deletions with red strikethrough, similar to track-changes in Google Docs; each suggestion block has accept/reject buttons)
3. Humans can accept/reject each agent suggestion with one click
4. Export to clean markdown file

**What to skip:** Auth, permissions, Confluence sync, file management, formatting toolbar, deployment polish, mobile, comprehensive tests.

**Prerequisite spikes (1 week before main build):**
1. Server-side Yjs participation: can a headless Node.js process reliably join a Yjs room, read document state, and apply edits using `y-protocols` directly? Spike this first as it is the highest-risk component.
2. Suggestion layer: can we implement "proposed changes" (accept/reject) on top of Yjs? Investigate Tiptap's built-in track-changes or a separate Yjs sub-document approach. **Fallback:** if the suggestion layer proves too complex, the MVP falls back to agent edits appearing as normal CRDT operations with a distinct purple cursor color (no accept/reject). This still validates the core "AI as peer collaborator" hypothesis.

The 2-4 week effort estimate assumes both spikes resolve favorably. Add 1 week for spike phase.

**Persistence:** Documents are stored server-side in a SQLite database keyed by a shareable URL slug. No file management UI; documents are accessed by direct link.

**Degradation behavior:** The editor continues working as a plain collaborative markdown editor when the AI API is unavailable. WebSocket reconnection with exponential backoff. Agent responses timeout after 30 seconds with a user-visible "Agent is taking longer than expected" indicator.

**Conflict behavior:** Human edits always take priority over pending agent suggestions. If a human edits a range that overlaps with an in-flight suggestion, the suggestion is auto-invalidated and removed. Accepting a suggestion applies the diff as a normal CRDT edit attributed to the agent; rejecting deletes the suggestion node. Both are undoable.

**Agent connection lifecycle:** Clicking "Invite Agent" sends a POST to the server, which spawns a long-lived worker (one per agent per document) that connects to the Yjs room via WebSocket. Workers are stateless beyond the Yjs connection and can be restarted without data loss because CRDT state is held in the Yjs room.

Main editor pane with rendered markdown (headings, paragraphs). Jason's orange cursor label showing real-time co-editing. ## Open Questions

1. **Architecture decision:** Should the document model be AI-native first or human-first? Concretely: "AI-native first" means the canonical model is a machine-readable CRDT changelog with markdown serialization as a derived view, and agents interact at the CRDT protocol level. "Human-first" means the canonical model is Tiptap's ProseMirror document, and AI accesses it through the same APIs as human browser clients. The subagent recommends AI-native first. This needs a deliberate decision before writing code.

2. **Agent interaction model:** Should agents only suggest edits (passive, human-approved) or also be able to directly edit (active, with undo)? MVP should start with suggestions only.

3. **Multi-agent support:** Can multiple AI agents collaborate on the same doc simultaneously? Defer to post-MVP but design the CRDT layer to support it.

4. **Enterprise GTM timing:** When does Confluence sync become necessary? After prototype validation with 10 engineering leads? After first paying customer?

5. **Competitive moat:** If ColaMD or incumbents add peer-style AI collaboration, what's the defensible position? Speed of iteration? Enterprise compliance features? Superior agent-human UX?

6. **Server-side CRDT participation (highest technical risk):** Having a headless Node.js process write into a Yjs document while preserving correct CRDT state requires using `y-protocols` directly or a server-side Yjs document instance (no browser, no Tiptap). This is the hardest engineering problem in the MVP. Spike this first.

7. **Suggestion layer on top of CRDT:** Standard Yjs has no concept of "proposed changes" vs. "committed changes." The accept/reject model requires a custom layer. Investigate whether Tiptap's built-in suggestion/comment features can be adapted, or whether a separate suggestion document is needed. This is the second-hardest technical problem.

## Success Criteria

**Pre-build (mockup validation):**
- Show the mockup to 3+ engineering leads at Confluence-using companies
- At least 1 says "when can I have it?" or describes how they'd use it (not just "interesting")
- Identify the #1 feature they'd need before trialing it with their team

**Post-prototype (working product validation):**
- 10 engineering leads from different companies try the prototype and can articulate why it's different from copy-pasting ChatGPT
- At least 3 engineering leads say they would use this in their team's daily workflow
- The accept/reject interaction on AI suggestions feels faster and more natural than any existing workflow
- Real-time collaboration between 2+ humans and 1+ AI agent works without sync conflicts

## Distribution Plan

- Public demo instance for try-before-you-buy
- Self-hosted option for enterprise compliance requirements (later)
- GitHub repo for open-core model (editor OSS, enterprise features paid)

- No blockers for MVP
